统一抽象语法树

统一抽象语法树

  统一抽象语法树(Unified Abstract Syntax Tree)简称UAST。

  UAST（统一抽象语法树）是针对JVM（Java虚拟机）的不同编程语言PSI上的一个抽象层。它提供了一个统一的API，用于处理常见的语言元素，如类和方法声明、字面价值和控制流操作符。

  不同的JVM语言有自己的PSI，但许多IDE功能，如检查、沟槽标记、引用注入和其他许多功能对所有这些语言的工作方式是一样的。使用UAST允许提供的功能将在所有支持的JVM语言中使用一个单一的实现。

  “为Kotlin编写IntelliJ插件的演示”提供了在真实场景中使用UAST的全面概述。

我在什么时候应该使用 UAST？

  对于插件，它应该以相同的方式适用于所有 JVM 语言。

  一些已知的例子有：

  • Spring框架
  • Android Studio
  • DevKit插件

支持哪些语言？

  • Java：完全支持
  • Kotlin：全面支持
  • Scala:测试版，但完全支持
  • Groovy：只支持声明，不支持方法体。

那么修改PSI呢？

  UAST是一个只读的API。有实验性的UastCodeGenerationPlugin和JvmElementActionsFactory类，但它们目前不推荐用于外部使用。

使用UAST工作

  UAST的基础元素是UElement。所有通用的基础子接口都位于uast模块的declarations和expressions目录中。

  所有这些子接口都提供获取有关常见语法元素信息的方法：UClass关于类声明，UIfExpression关于条件表达式等等。

PSI到UAST的转换

  要获取支持语言之一的给定PsiElement的UAST，请使用UastFacade类或UastContextKt.toUElement()：

Java

UastContextKt.toUElement(element);

Kotlin

element.toUElement()

  要将PsiElement转换为特定的UElement，请使用以下方法之一：

  • 对于简单的转换：

Java

UastContextKt.toUElement(element, UCallExpression.class);

Kotlin

element.toUElement(UCallExpression::class.java)
  • 用于转换为给定不同选项之一：

Java

UastFacade.INSTANCE.convertElementWithParent(element,
    new Class[]{UInjectionHost.class, UReferenceExpression.class});

Kotlin

UastFacade.convertElementWithParent(element,
    UInjectionHost::class.java, UReferenceExpression::class.java)
  • 在某些情况下，PsiElement 可能表示几个 UElement。例如，Kotlin 主构造函数的参数同时是 UField 和 UParameter。当需要所有选项时，请使用：

Java

UastFacade.INSTANCE.convertToAlternatives(element,
    new Class[]{UField.class, UParameter.class});

Kotlin

UastFacade.convertToAlternatives(element,
    UField::class.java, UParameter::class.java)

最好是转换为UElement的具体类型，而不是在没有类型的情况下进行转换，然后再转为具体类型：

  • 因为性能原因：toUElement()在类型检查时会快速失败。
  • 由于在某些情况下可能会得到不同的结果，因此使用类型进行转换更具可预测性。
UAST到PSI的转换

  有时需要从 UElement 返回到底层语言的源代码。为此，UElement＃sourcePsi 属性返回原始语言中相应的 PsiElement。

  sourcePsi是一个“物理”的PsiElement，主要用于获取原始文件中的文本范围（例如，用于高亮显示）。避免将sourcePsi转换为特定的类，因为这意味着从UAST抽象回退到特定于语言的PSI。一些UElement是“虚拟”的，因此没有sourcePsi。对于某些UElement，sourcePsi可能与获取UElement的元素不同。

  此外，还有一个UElement#javaPsi属性，可以返回一个 "类似Java "的PsiElement。这是一个 "假的 "PsiElement，使不同的JVM语言模拟Java语言，以保持与Java-API的兼容性。例如，当调用MethodReferencesSearch.search(PsiMethod)时，只有Java原生提供PsiMethod；其他JVM语言因此通过UMethod#javaPsi提供一个 "假的 "PsiMethod。

  请注意，UElement＃javaPsi仅适用于Java。因此，应使用UElement＃sourcePsi以获取文本范围或锚定元素，以进行检查警告/差错标记放置。

  简言之：

  • sourcePsi:

    • 是物理的：在原始语言的源代码中表示一个真实存在的PsiElement

    • 可用于高亮显示、PSI修改、创建智能指针等。

    • 除非绝对必要（例如处理特定语言情况），否则不应强制转换。

  • javaPsi

    • 应该将其仅用作JVM可见声明的表示：PsiClass、PsiMethod、PsiField用于获取它们的名称、类型、参数等，或将它们传递给接受Java-PSI声明的方法。
    • 不能保证是物理存在的：可能不存在于源数据中。
    • 不可修改：对于非Java语言，调用修改方法可能会引发异常。

  注意：sourcePsi和javaPsi都可以转换回UElement。

UAST访问者

  在UAST中，没有一种统一的方法可以获取UElement的子元素（尽管可以通过UElement#uastParent获取其父元素）。因此，以树形式遍历UAST的唯一方法是将UastVisitor传递给UElement.accept()方法。

  注：在UAST-visitors中有一个约定，如果visit*()返回true，则不会将访问者传递给子节点。否则，UastVisitor将继续深入遍历。

  可以使用 UastVisitorAdapter 或 UastHintedVisitorAdapter 将 UastVisitor 转换为 PsiElementVisitor。后者是更可取的，因为它提供更好的性能和更可预测的结果。

  一般规则是建议避免使用UastVisitor：如果您不需要处理许多不同类型的UElement，并且元素的结构不是非常重要，那么最好使用PsiElementVisitor遍历PSI-tree，并使用UastContext.toUElement()将每个PsiElement明确地转换为其对应的UAST。

UAST性能提示

  UAST 并不是一种零成本的抽象：对某些语言而言，某些方法可能会意外地变得昂贵，因此在优化时要小心，因为这可能会产生相反的效果。

  在某些情况下，转换到UElement也可能需要对某些语言进行解析，同样，可能会有意想不到的代价。转换到UAST应该只在必要时进行。例如，将整个PsiFile转换为UFile，然后仅仅为了收集UMethod的声明而走一遍是低效的。相反，遍历PsiFile并将每个遇到的匹配元素显式地转换为UMethod。

  当你把访问者传给UElement.accept()或得到UElement#uastParent时，UAST是懒惰的。

  对于真正的性能优化，请考虑使用UastLanguagePlugin.getPossiblePsiSourceTypes()在将PsiElement转换为UAST之前进行预过滤。

UAST注意事项

ULiteralExpression 不应该用于字符串。

  ULiteralExpression表示字面量值，如数字、布尔值和字符串。虽然字符串值也是字面量，但使用ULiteralExpression来处理它们并不方便。例如，它不能处理Kotlin的字符串插值。如果要在评估它们的值时处理字符串字面量或执行语言插入，应该使用UInjectionHost。

sourcePsi和javaPsi、psi和UElement作为PSI

  由于历史原因，UElement和PsiElement之间的关系很复杂。一些UElements实现了PsiElement；例如，UMethod实现了PsiMethod。我们强烈反对将UElement作为PsiElement使用，Plugin DevKit提供了相应的检查（Plugin DevKit | Code | UElement as PsiElement usage）。这个 "实现 "被认为是废弃的，将来可能会被删除。

  另外，还有UElement#psi属性；它返回与javaPsi或sourcePsi相同的元素。由于很难猜测会返回什么，因此它也已经过时。

  因此，sourcePsi 和 javaPsi 应该是从 UElement 获取 PsiElement 的唯一方式。请参见对应的章节。

使用UAST或PSI

  UAST提供一种统一的方式，通过UMethod、UField、UClass等来表示与JVM兼容的声明。但同时，所有的JVM语言插件都会实现PsiMethod、PsiClass等来与Java兼容。这些实现可以通过UElement#javaPsi属性获得。

  因此问题是："我应该在我的代码中使用什么来表示Java声明？"。答案是：我们鼓励使用PsiMethod、PsiClass作为Java声明的常见接口，而不管JVM语言，并且不鼓励在API中暴露UAST接口。

  注意：对于方法体来说，没有这样的选择，所以不鼓励暴露，例如，UExpression。不过，可以考虑用原始的PsiElement来代替。

UAST/PSI树结构不匹配

  UAST是在不同语言的PSI之上的一个抽象层次，它试图建立一个统一的树（见检查UAST树）。这导致了UAST和原始语言之间的树形结构可能严重分歧，所以不能保证保留祖先-后裔关系。

  例如，以下的结果：

generateSequence(uElement, UElement::uastParent).mapNotNull { it.sourcePsi }
generateSequence(uElement.sourcePsi) { it.parent }

  可能是不同的，不仅在元素数量上，还可能在它们的顺序上不同。

在插件中使用UAST

  要在您的插件中使用UAST，请添加一个依赖关系到捆绑的Java插件（com.intellij.java）。

语言扩展

  要注册适用于UAST的扩展程序，需要在plugin.xml中的注册中指定language="UAST"。

检查UAST树

  要检查UAST树，请调用内部动作Tools | Internal Actions | UAST | Dump UAST Tree（按每个Psi元素）。

检查

  使用AbstractBaseUastLocalInspectionTool作为基类，并在注册中指定language="UAST"。如果检查目标仅针对默认类型的子集（UFile、UClass、UField和UMethod），则在重载构造函数中指定UElements作为提示以提高性能。

线条标记

  使用UastUtils.getUParentForIdentifier()或UAnnotationUtils.getIdentifierAnnotationOwner()获取用于注释的适当的"标识符"元素（有关详细信息，请参阅行标记提供者）。
